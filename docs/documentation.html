1 · What it does

CP Intelligence turns the 15–20 day manual hunt for channel partners into a ranked, scored shortlist on day one.

For each launch (a project a developer is selling), a Sales Manager can:

  • Match verified CPs against the launch criteria (location, configuration, price band, developer type) and see them scored 0–100 with an explainable breakdown.
  • Triage them on a Proven × Fit quadrant.
  • Study the micro-market — nearby competing projects, real transacted prices, and where matched CPs already sell — on a live map.
  • Shortlist partners into custom categories and track each from New → Contacted → Meeting → Proposal → Tie-up Done, ending with a Final Launch Partners list.
Access: open the live app and click Sign in → (no credentials — authentication is stubbed for the demo).

2 · User guide — screen by screen

Sidebar: Dashboard · Launches · CP Directory · Launch Match · Micro-Market · Shortlist. The footer shows the current launch and a quick "N in shortlist" link.

Login

Product intro + Sign in →. Clicking sign-in enters the app (no real auth).

Dashboard

Data-health KPIs — verified %, RERA/PAN/GST coverage, recent refreshes — a snapshot of how current and trustworthy the CP database is.

Launches

  • Portfolio list with status pills Active Draft Completed Inactive, a summary strip, and filter chips.
  • Each row shows a shortlist badge 📌 N shortlisted · 🤝 M finalized — click it to open that launch's Shortlist board.
  • Status actions: Micro-Market, Activate, Complete / Archive, Reopen.
  • + New Launch drawer includes a free, keyless address autocomplete (OpenStreetMap/Photon): type an address, pick a suggestion, and it auto-fills Street, Country, State, District, PIN (all editable) — plus BHK, price band, developer tier, status, units, notes.

Launch Match

  • Criteria: all-India State → District → City cascade, PIN chips, BHK chips, price band, Verified-only, and Advanced filters = 6 weight sliders (Geography, Configuration, Price-band, Developer-type, Recency, Compliance) that re-score and re-rank every CP live.
  • Summary widget: Matches · Shortlisted · Contacted · Meetings · Finalised.
  • Tab · Matching Channel Partners: CPs grouped into the 4 quadrant categories with counts; each row has the score, a WHY? explainer, a Verified badge, and Call / Email / Tie up / Shortlist. Overflow behind "+N more" → multi-select bulk popup.
  • Tab · Quadrant View: the same filtered CPs on a 2×2 Proven × Fit grid + an Approach-first list. Both tabs always show the same set.

CP Directory

Browse the full database — search (firm / XR), quadrant filter, pagination, per-row Call / Email / View Profile, and Corrections tracking (System vs User source).

Profile 360

Full CP dossier: score ring + breakdown, compliance & documentation (RERA/PAN/GST), performance (avg VDNB, units sold), market coverage, track record, developers worked with, tied launches, and quick actions.

Micro-Market

  • Live Leaflet + OpenStreetMap map with tower markers coloured by category; the launch is a larger purple tower.
  • The launch label sits to the right of its marker and auto-flips left near the edge so it's never clipped.
  • Nearby competitor projects side panel (name, developer, distance, category, config, price, status) — two-way linked with the map markers.
  • KPI bar, asking-vs-transacted chart, and a competitor-CP map (who already sells nearby + leak risk).

Shortlist board ★

  • Custom categories per launch — create / rename / delete / reorder (⚙ Manage categories), each with a CP count.
  • Add to Shortlist popup — a multi-select launch dropdown (add one CP to several launches at once) + category picker + Create new category in one step.
  • Progress — status dropdown (New · Contacted · Meeting Scheduled · Meeting Done · Proposal Shared · Negotiation · Tie-up Done · Rejected) + one-tap quick actions and Move to….
  • Filters — status chips + Verified-only, Category, City, Assigned user, Sort.
  • Final Launch Partners — CPs marked Tie-up Done collect automatically, with a total.
  • Timeline — dated activity log per CP; Summary widget across all stages.

4 · Technical reference

Architecture

  • The whole app is one file — CP Intelligence.dc.html — authored in the Claude Design runtime.
  • support.js self-loads React 18 + Babel and fonts from CDN and calls fetch(location.href) — so the file must be served over HTTP (not file://) and needs internet on first paint.
  • Templating: <sc-if>, <sc-for>, {{bindings}}; a class extends DCLogic with state + renderVals(). ScoreRing.dc.html is an imported sub-component.

Data model (in-memory, seeded)

  • cpData — CP records; extended by buildCitySeeds() (13 cities) and on-demand ensureCitySeed() so Launch Match is never empty.
  • geoTree — { state:{ district:{ city:[pins] }}}, 36 states/UTs · 662 districts · ~15,500 cities with real PINs (GeoNames-derived).
  • launches — ~12 launch records.
  • Shortlist store — sl[launchId] = { cats:[…], items:{ [xr]:{cat,status,assigned,added,log} } }; SL_STATUS = 8 statuses.
  • Scoring — weights[6] / tunable scoreWeights[6] / dynScore(cp) / makeBreakdown(); verifiedSet drives the verified badge.

External services (all keyless / free)

UseService

No API keys, no billing — deliberate, since the site is public. Best-effort public endpoints; if unreachable, fields stay manual.

Repository / file map

  • project/ — working copy (app, support.js, ScoreRing.dc.html).
  • site/ — the deployed GitHub repo: app at the repo root + a byte-identical docs/ mirror + .nojekyll. Six HTML copies kept in md5 parity on every change.
  • server.js, package.json, share.js — local runner + sharing.

# change port:  PORT=8080 npm start   (PowerShell: $env:PORT=8080; npm start)

npm run share      # server + Cloudflare quick-tunnel → prints a public https URL
npm run tunnel     # localtunnel on :5173

Same wifi: npm start also prints a LAN URL (http://<ip>:5173).

Needs internet — React/Babel/fonts, map tiles and geocoders all load from the network.

6 · Deployment

  • Hosted on GitHub Pages, repo Anjalijha2/cp-intelligence, branch main.
  • Pages serves the repo root; a docs/ mirror is kept identical; .nojekyll disables Jekyll (the file is {{…}}-heavy). Publishing uses the "pages build and deployment" Action.

Ops notes

  • After editing the app, sync all 6 HTML copies and confirm md5 parity = 1 before committing.
  • A Pages build can occasionally be cancelled (runner/concurrency) → it serves the previous build. Fix: push again to re-trigger.
  • Pages throttles to ~10 builds/hour; batch changes to avoid queueing.
  • Confirm the deploy via the Actions API (completed/success) before assuming the live site updated; then hard-refresh (Ctrl+Shift+R).

7 · Known limitations

A front-end prototype for demos, not production:

  • No backend / persistence — all state is in memory and resets on reload (shortlists, statuses, created launches, categories).
  • No real authentication — sign-in is a stub.
  • Illustrative data — CP records, prices, RERA/PAN/GST and scores are fabricated.
  • Detail screens are tuned around the Borivali West demo launch; others reuse the demo terrain.
  • CPs are seeded for known cities; other locations get generic auto-seeded CPs so no view is empty.
  • Geocoding/autocomplete use free public endpoints with no uptime guarantee.
